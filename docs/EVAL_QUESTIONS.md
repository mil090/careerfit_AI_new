# EVAL_QUESTIONS.md — CareerFit AI 자기 평가 질문



## 이 질문들에 스스로 답할 수 있어야 최종 발표와 면접에 나설 수 있다.



## 프로젝트 전체

- Q1. 이 서비스가 해결하는 문제는 무엇인가?
- A1. 이 서비스는 사용자의 정보(전공, 보유 스킬, 관심 직무 등)를 입력받아 취업에 도움이 되는 맞춤 정보를 제공하는 역할을 한다.

- Q2. RAG를 사용하지 않으면 어떤 문제가 생기는가?
- A2. RAG를 사용하지 않으면, 언어 모형(LLM)이 실제 취업 공고 데이터가 아니라 자신의 학습된 지식만을 사용하기 때문에, 근거가 없는 정보를 생산하여 환각(Hallucination)을 일으킬 가능성이 커진다.

- Q3. 가장 어려웠던 구현은 무엇이고, 어떻게 해결했는가?
- A3. GEMINI, ollama 등의 언어 모형을 다운로드하고, 이 모형들을 서버에 연결하는 과정이 가장 어려웠다. 연결 과정 자체는 크게 어려운 점이 없었으나, 코드 작성 과정에서 변수명을 다르게 설정한 탓에 언어 모형이 서버에 제대로 연결되지 않았고, 이를 찾는 데 오랜 시간이 걸렸다. Careerfit AI 조교를 통해 어디에서 변수명이 서로 다르게 작성되었는지 찾아 이를 바로잡을 수 있었다.



## 데이터 관련

- Q1. 왜 이 데이터를 선택했는가?
- A1. 다양한 취업 공고를 참고함으로써 어떤 기업이 어떤 역량을 요구하는지 파악하여 사용자의 필요에 맞는 정보를 선별하는 데 도움이 될 것이라고 판단하여 이 데이터(jobs.csv)를 선택하였다.

- Q2. 데이터의 어떤 부분이 신뢰할 수 없을 수 있는가?
- A2. 데이터에서 소개되는 업무 내용이 실제와 차이가 있을 수 있다. 공고에서 소개되는 업무는 비교적 간단히 서술되어 있으므로, 실제 업무는 이와 완전히 같지는 않을 수 있을 것으로 보인다.

- Q3. SQLite와 ChromaDB에 같은 데이터를 저장하는 이유는 무엇인가?
- A3. 두 데이터베이스에 같은 데이터를 저장하는 이유는 두 데이터베이스의 강점이 서로 다르기 때문이다. SQLite는 정확한 조건 탐색을, ChromaDB는 의미 기반 탐색(RAG)을 담당한다.

- Q4. 결측치와 중복을 어떻게 처리했는가?
- A4. 결측치와 중복값은 모두 Python 함수를 통한 전처리 과정으로 처리하였다. 결측치가 존재할 경우 그 행이 어떤 행이며 총 몇 개의 행에 결측치가 있는지 표시하도록 하였으며, 중복값이 발견될 경우 처음으로 나온 행을 제외하고 모두 제거하도록 구현하였다.



## RAG 관련

- Q1. RAG 구조에서 ChromaDB는 어떤 역할을 하는가?
- A1. ChromaDB는 RAG에서 사용자 질문과 의미가 비슷한 문서를 찾아 이를 언어 모형에 전달하는 역할을 한다. 즉 답변의 근거를 탐색하는 역할이라고 할 수 있다.

- Q2. `n_results` 값을 높이면 어떤 트레이드오프가 생기는가?
- A2. 결과 수(`n_results`)가 늘어나면 더 많은 근거 자료를 활용할 수 있으나 상대적으로 관련성이 떨어지는 자료까지 포함될 수 있다는 트레이드오프가 발생한다.

- Q3. 검색 결과 품질을 어떻게 검증했는가?
- A3. ChromaDB의 `distance` 값을 통해 질문과 검색 결과의 관련성을 확인하였다.

- Q4. Hallucination이 무엇이며, RAG가 이를 어떻게 줄이는가?
- A4. 환각(Hallucination)이란 언어 모형이 거짓 내용을 그럴듯하게 만들어내는 현상을 말한다. RAG는 검색된 실제 데이터를 함께 제공하여 이러한 현상을 줄인다.

## API 관련

- Q1. `/analyze` 엔드포인트가 POST를 사용하는 이유는 무엇인가?
- A1. `/analyze`는 여러 입력 데이터를 요청 본문(Body)으로 받아 분석 결과를 생성하는 작업이기 때문이다.

- Q2. 응답 JSON에 `sources`를 포함시킨 이유는 무엇인가?
- A2. AI가 어떤 데이터를 근거로 답변했는지 사용자에게 보여주어 답변의 신뢰성과 검증 가능성을 높이기 위함이다.

- Q3. `/health` 엔드포인트가 왜 필요한가?
- A3. 서버가 정상적으로 작동하고 있는지 빠르게 확인하기 위함이다.



## 보안 관련

- Q1. `.env` 파일을 GitHub에 올리면 어떤 일이 생기는가?
- A1. `.env` 파일을 GitHub에 올리면 API Key와 같은 민감한 정보가 공개되어, 다른 사람이 이를 사용할 수 있고 서비스가 악용될 위험이 높아진다.

- Q2. `.env`와 `.env.example`의 차이는 무엇인가?
- A2. `.env`는 실제 비밀 정보를 저장하는 파일이고, `.env.example`은 어떤 환경변수가 필요한지 알려주는 템플릿 파일이다. `.env`와 달리 `.env.example`은 실제 API Key를 포함하지 않기 때문에 GitHub에 자유롭게 업로드할 수 있다.



## Docker 관련

- Q1. Docker를 사용하는 이유는 무엇인가?
- A1. 개발 환경을 하나의 컨테이너로 묶어 어떤 컴퓨터에서 실행해도 동일한 결과를 보장하기 위해서이다.

- Q2. `Dockerfile`에서 `EXPOSE`와 `CMD`의 역할은 무엇인가?
- A2. `EXPOSE`는 Docker가 사용할 포트를 알리는 역할이고, `CMD`는 Docker가 시작될 때 실행할 기본 명령어를 지정하는 역할이다.
